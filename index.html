O que é Git?

Git é um Sistema de Controle de Versão Distribuído. Originado do desenvolvimento do kernel Linux (Linus Torvalds) e é utilizado por muitos projetos Open Source. O core do Git foi originalmente escrito em C, no entanto já foi reimplementado em outras linguagens como Java, Ruby e Python.

O que é um Sistema de Controle de Versão? E por que devo me importar com isso?

Controle de versão é um sistema que registra as mudanças feitas em um arquivo ou um conjunto de arquivos ao longo do tempo de forma que você possa recuperar versões específicas, você pode usá-lo com praticamente qualquer tipo de arquivo em um computador. Ele permite reverter arquivos para um estado anterior, reverter um projeto inteiro para um estado anterior, comparar mudanças feitas ao decorrer do tempo, ver quem foi o último a modificar algo que pode estar causando problemas, quem introduziu um bug e quando, e muito mais. Usar um VCS normalmente significa que se você estragou algo ou perdeu arquivos, poderá facilmente reavê-los. Além disso, você pode controlar tudo sem maiores esforços. Git é um sistema de controle de versão.

Sistemas de Controle de Versão Locais

O método preferido de controle de versão por muitas pessoas é copiar arquivos em outro diretório. Esta abordagem é muito comum por ser tão simples, mas é também muito suscetível a erros. É fácil esquecer em qual diretório você está e gravar acidentalmente no arquivo errado ou sobrescrever arquivos sem querer.

Sistemas de Controle de Versão Centralizados

Outro problema está na necessidade de trabalhar em conjunto com outros desenvolvedores que usam outros sistemas. Para lidar com isso, foram desenvolvidos Sistemas de Controle de Versão Centralizados. Esses sistemas, como por exemplo o Subversion, possuem um único servidor central que contém todos os arquivos versionados.

Sistemas de Controle de Versão Distribuídos

Da necessidade de solução para as questões anteriores, surgem os Sistemas de Controle de Versão Distribuídos. Nestes sistemas os clientes são cópias completas do repositório. Assim, se um servidor falha, qualquer um dos repositórios dos clientes pode ser copiado de volta para o servidor para restaurá-lo.

Quem usa?

Ok, mas... e Github?! O que raios é isso?

GitHub é um Serviço de Web Hosting Compartilhado para projetos que usam o controle de versionamento Git. É escrito em Ruby on Rails pelos desenvolvedores da Logical Awesome (Chris Wanstrath, PJ Hyett e Tom Preston - Wernder). O GitHub possui planos comerciais e gratuitos para projetos de código aberto. Este site possui funcionalidades de uma rede social como feeds, followers, wiki e um gráfico que mostra como os desenvolvedores trabalham as versões de seus repositórios. O GitHub também possui um serviço tipo pastebin chamado Gist.

Repositório local

É o repositório que se cria ou se clona localmente. Após clonar ou criar, o usuário possui uma cópia completa do repositório e além diso executa operações de controle de versão neste repositório, cria novas versões, desfaz alterações, etc. Se você quiser excluir um repositório local, basta excluir a pasta do mesmo. O repositório local nada mais é do que uma pasta com os arquivos que se deseja versionar.

Repositório remote

É o repositório que está localizado no servidor. Usuários com a devida autorização (autenticação) podem realizar mudanças no seu repositório local e enviá-las ao repositório remote. Também podem sobrescrever e atualizar (tracking) seus repositórios locais a partir do repositório remote.

Como as coisas acontecem...

Branches

Branches ("ramos, braços") são utilizados para desenvolver funcionalidades isoladas umas das outras. O branch master é o branch "padrão" quando você cria um repositório. É recomendado utilizar e criar branches separados para desenvolver uma nova funcionalidade para então mesclá-lo (merge) ao branch master após a conclusão.

Para verificar quais branchs ativos você possui é só utilizar o comando 

$git branch

  * gh-pages
    master

Remover um branch localmente e remotamente.

localmente: 

$git branch -D [nomebranch]
remotamente: 
$git push [nomerepo] --delete [branch]

Trocar de branch!

Para trocar de um branch para outro que se deseje trabalhar usamos o comando: 

$git checkout [nomebranch]

Switched to branch '[nomebranch]'

adicionando mudanças ao staging 

$git add [nome_arquivo]

Por favor! Status!

O principal comando utilizado para determinar quais arquivos estão em quais estados é o comando git status. 

$git status
Vamos dizer que você adicione um novo arquivo em seu projeto, um simples arquivo README. Se você executar git status, você verá o arquivo não monitorado dessa forma: 
      $ touch README.txt
      # On branch master
      # Untracked files:
      # (use "git add ..." to include in what will be committed)
      #
      #  README
      nothing added to commit but untracked files present (use "git add"
      to track)

commit

É o versionamento propriamente dito. Consiste em fazer uma confirmação de armazenamento do estado atual dos arquivos localizados no seu repositório local. Cada commit possui uma identificação única e pertence a um único branch, ou seja, o branch atual de trabalho. É interessante fazer analogia a uma "foto" do repositório em um determinado momento.

e para commitar... 

$git commit -m 'Seu commit aqui'

Visualizar o histórico de Commits

Depois que você tenha criado vários commits, ou se você clonou um repositório com um histórico de commits existente, você provavelmente irá querer ver o histórico. O comando para fazer isso é: 

$git log

Push

É o comando de subir ou enviar uma alteração ao servidor que hospeda o repositório remote. Mantém um branch específico do repositório remote localizado no servidor sincronizado com as alterações locais.

$git push [-opções] [nome_repo] [nome_branch]

Pull

É o comando de descer ou atualizar o seu repositório local a partir do servidor que hospeda o repositório. Mantém o repositório local sincronizado com todos os commits de um branch específico a partir do repositório remote. 

$git pull [nome_repo] [nome_branch]

Para melhorar o entendimento...

Merge

É a junção de dois branches para um mesmo ponto. Suponha que você decidiu que o trabalho está completo. Para fazer isso, você fará o merge do seu branch em questão com a nova funcionalidade, Tudo que você tem a fazer é executar o checkout do branch para onde deseja fazer o merge e então rodar o comando git merge: 

$git merge [nome_do_repo_com_nova_funcionalidade]

Clone

Caso você queira copiar um repositório Git já existente o comando necessário é: 

$git clone [url]

Git recebe uma cópia de quase todos os dados que o servidor possui. Cada versão de cada arquivo no histórico do projeto é obtida quando você roda git clone. Você clona um repositório com:

Clonando...

Por exemplo, caso você queira clonar os arquivos da apresentação, poderá fazê-lo da seguinte forma: 

$git clone https://github.com/pvsousalima/grupython_apresentacao.git

Isso cria um diretório local controle-medicamentos, inicializa um diretório .git dentro deste, obtém todos os dados do repositório e verifica a cópia atual da última versão. Se você entrar no novo diretório criado verá todos os arquivos do projeto nele, prontos para serem editados ou utilizados. O Git possui diversos protocolos de transferência que você pode utilizar. O exemplo anterior utiliza o protocolo https://, mas você também pode ver git(s):// ou protocolo de transferência SSH.

Fork

Fork é uma opção presente no github. Ao se fazer uso dessa opção você clona remotamente um repositório pré-existente que queira contribuir para o seu grupo de repositórios.

Pull-request

É a ferramenta que te permite realmente contribuir com algum projeto! A partir de algum repositório que foi clonado usando o comando fork visto anteriormente, você pode realizar as suas modificações no seu repositório clonado e a partir destas modificações solicitar ao dono do repositório original que ele aceite as modificações no projeto dele feitas a partir do seu repositório.
